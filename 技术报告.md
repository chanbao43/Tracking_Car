# 技术报告 — 2路红外循迹小车

## 项目背景

循迹小车是嵌入式系统与智能控制领域的经典实战项目，通过传感器实时感知地面轨迹，经单片机处理后控制电机完成自主行驶。本项目以 STC89C52RC 单片机为核心，搭载两个红外传感器、两个直流减速电机及 L298N 驱动模块，实现了一台能够在两圈复杂赛道（含 180°掉头与 90°直角转弯）上自主循迹的智能小车。

项目从一段基础参考代码出发，经历了三次架构级重构：初版采用阻塞式延时和直接在中断中处理逻辑的原始方案；第二版引入任务调度器概念但仍有缺陷；最终版（当前 main.c）实现了**单定时器 + 任务调度器**架构，ISR 极简化，所有决策逻辑移至 main() 调度循环，配合状态机、去抖算法和非阻塞延时，彻底解决了旧版在复杂赛道上的稳定性问题。

## 需求分析

### 功能需求

| 编号 | 功能 | 说明 |
|------|------|------|
| F1 | 红外循迹 | 2路传感器实时检测黑线，支持直行/左转/右转纠偏 |
| F2 | PWM 调速 | 20级 PWM 控制电机转速，实现差速转向 |
| F3 | 线计数与去抖 | 准确统计经过的黑线数量，消除机械抖动误触发 |
| F4 | 180° 掉头 | 第一圈结束自动原地掉头，切换至第二圈 |
| F5 | 90° 转弯 | 第二圈指定位置自动完成 90° 直角转弯 |
| F6 | 两圈赛道管理 | 自动区分第一圈/第二圈，完成两圈后停车 |
| F7 | 数码管显示 | 实时显示当前经过的线数 |
| F8 | 按键启动 | 按下按键后小车启动，便于调试和比赛 |

### 非功能需求

- **实时性**：所有传感器响应和电机控制必须在 1ms 内完成一轮决策
- **可靠性**：线计数必须具备抗抖动能力，杜绝误计数和漏计数
- **可维护性**：代码模块化，每个任务函数职责单一，参数集中定义便于调参

## 关键代码实现说明

### 1. 单定时器 + 任务调度器架构

这是本项目最核心的设计决策。Timer 0 以 1ms 周期产生中断，ISR 内仅做三件事：

```c
void TIMER0_IRQHandler(void) interrupt 1 using 2
{
    // 1. PWM 生成（硬件无关的逻辑运算）
    Left_PWM_Value++; Right_PWM_Value++;
    Left_Motor_PWM_Adjust(); Right_Motor_PWM_Adjust();

    // 2. 非阻塞延时：软件定时器自增
    if(cooldown_active) cooldown_timer++;
    if(uturn_active)    uturn_timer++;
    if(turn90_active)   turn90_timer++;

    // 3. 系统时基标志
    sys_tick++;
    sys_tick_flag = 1;
}
```

main() 中的调度循环按优先级执行 7 个任务：

```c
while(1)
{
    if(!sys_tick_flag) continue;  // 等待 1ms 节拍
    sys_tick_flag = 0;

    Task_UTurn();         // 180°掉头（动作任务）
    Task_Turn90();        // 90°转弯（动作任务）
    Task_PhaseManager();  // 阶段管理
    Task_SensorMotor();   // 传感器 + 电机
    Task_LineCounter();   // 计线状态机
    Task_CooldownTimer(); // 冷却管理
    Task_Display();       // 数码管刷新
}
```

架构优势：ISR 极短（耗时 < 50μs）不会阻塞自身，所有复杂决策在主循环中完成，逻辑清晰易调试。

### 2. 状态机线计数（三重去抖 + 计数锁）

线计数是本项目最容易出问题的环节。红外传感器在经过黑线边缘时会产生大量毛刺信号（抖动），导致一次过线被多次计数。

**四级状态机设计：**

```
SM_WAIT_BLACK → SM_ON_BLACK → SM_WAIT_WHITE → SM_COUNTED
      ↑                                              │
      └────────── 冷却结束 ──────────────────────────┘
```

- `SM_WAIT_BLACK`：等待全白信号，连续 8ms 确认后才进入黑线状态（第一重去抖）
- `SM_ON_BLACK`：确认在黑线上，连续 8ms 非全白后才判定离开（第二重去抖）
- `SM_WAIT_WHITE`：等待传感器回到全黑信号，连续 40ms 确认后计线 +1（第三重去抖）
- `SM_COUNTED`：计线完成，等待 400ms 冷却后回到 `SM_WAIT_BLACK`

**计数锁机制：** 即使通过了三重去抖，如果两次计数间隔 < 380ms，系统会拒绝本次计数并重置状态机。这一方案彻底消除了因机械振动或赛道瑕疵导致的重复计数。

### 3. 非阻塞动作任务

传统方案中，180°掉头和 90°转弯使用 `delay_ms()` 阻塞实现，导致转弯期间传感器停止检测、小车"盲跑"。本项目将所有动作任务改造为非阻塞形式：

```c
void Task_UTurn(void)
{
    if(!uturn_active) {
        // 判断触发条件
        if(phase != FIRST_LAP || Line < 8 || task_busy) return;
        uturn_active = 1; uturn_timer = 0; task_busy = 1;
    }
    // 每 1ms 执行一次，不阻塞
    Left_Motor_Back; Right_Motor_Go;
    if(uturn_timer >= UTURN_DURATION) {
        Stop();
        uturn_active = 0; task_busy = 0;
        phase = SECOND_LAP; Line = 0;
    }
}
```

关键设计：用 `task_busy` 全局互斥锁保证同一时间只有一个动作任务执行，动作期间传感器和计线任务自动退出（`if(task_busy) return`）。

### 4. 90° 转弯多阶段状态机

90°转弯是技术难点——小车速度快，直接转弯容易因惯性偏离。本项目采用四阶段执行：

- **Phase 0（冲线）**：转弯前先短距离直行，确保完全越过黑线
- **Phase 1（转弯）**：原地左转 240ms
- **Phase 2（停车）**：停车 300ms，消除旋转惯性
- **Phase 3（稳定直行）**：直行 125ms 恢复姿态后交还传感器控制

## 出现问题及解决方案

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 经过一条黑线被计数多次 | 传感器在黑白交界处产生毛刺抖动 | 引入三重去抖状态机 + 380ms 计数锁 |
| 180° 掉头后线数跳变 | 掉头过程中传感器扫过黑线导致误计数 | 加入 `task_busy` 互斥锁，动作期间跳过计线任务 |
| 90° 转弯后小车偏离轨迹 | 旋转惯性导致车身未及时回正 | 增加 Phase 2 停车消惯性 + Phase 3 稳定直行 |
| 转弯后立即遇线被重复触发 | 同一条线两次触发转弯 | 加入 `last_turn_line` 防重入判断 |
| 数码管显示闪烁 | 主循环中存在阻塞式延时 | 全部改为非阻塞实现，数码管每 1ms 刷新一次 |
| 长时间运行后电机无力 | 电池电压下降，固定 PWM 占空比不再合适 | 保留调参接口（SPEED 等宏定义），可根据电池情况调整 |

## 总结

本项目从一段基础循迹代码出发，经过三轮架构迭代，最终实现了一台稳定可靠的两圈循迹小车。核心收获如下：

**1. 架构设计方面**：任务调度器架构将 ISR 从繁重的业务逻辑中解放出来，提升了系统实时性和可维护性。这一思想在 RTOS 中普遍应用，本项目在裸机上手动实现了其简化版本。

**2. 状态机思维**：线计数和转弯控制通过状态机建模后，复杂的时序逻辑变得清晰可控。每一步状态转换都有明确的触发条件和去抖机制，调试时只需跟踪状态变量即可快速定位问题。

**3. 工程实践方面**：分步测试策略（7 个测试版本渐进验证）有效降低了调试复杂度。每个测试版本只验证单一功能，通过后才叠加新功能，避免了"一次写完、全面崩溃"的困境。

**4. 局限性**：当前小车仅支持固定赛道（8线/圈），参数（如转弯时长、去抖阈值）需针对特定赛道和电池电压手工调整，尚未实现自适应。此外，2 路传感器的信息量有限，仅能判断"偏左/偏右"，无法实现更精细的 PID 控制或前瞻控制。这些可作为未来改进方向。

通过本项目，我对嵌入式系统中的实时调度、状态机设计、传感器去抖和电机控制有了深入理解，为后续更复杂的嵌入式项目打下了基础。
